# DESIGN-20260722-01-文档治理体系改造方案

## 关联需求

- `docs/requirements/REQ-20260722-01-文档治理体系改造.md`

## 当前现状

- `AGENTS.md` 内容过重，混合了 Agent 行为规则和项目细节。
- `docs/` 已有接口文档和 Vue 工程化生成对接说明，但缺少统一治理规范。
- 用户的实际工作流是“给需求 -> AI 深挖 -> 商榷 -> 清晰方案 -> AI 执行 -> 用户审核 -> AI 修复/用户提交 -> 总结笔记”，适合拆成轻量可追溯文档。

## 设计方案

采用五类文档：

| 类型 | 作用 |
| --- | --- |
| `REQ` | 固化需求、范围、验收标准 |
| `DESIGN` | 固化方案、链路、取舍、验证计划 |
| `BUG` | 固化问题现象、根因、修复、验证 |
| `PROG` | 固化每日进度、证据、阻塞和下一步 |
| `NOTE` | 固化学习总结、链路解释和复盘 |

`AGENTS.md` 只保留行为规则：

- 文档读取顺序
- 任务开始前检查项
- 前后端代码边界
- AI 生成链路边界
- 验证和交付标准
- Git 与文件安全
- Windows 与编码注意事项

## 文档落点

```txt
docs/
  requirements/
  designs/
  bugs/
  progress/
  notes/
  文档治理规范.md
  项目总览.md
  本地运行与验证.md
  后端架构与开发规范.md
  前端架构与开发规范.md
  AI生成链路与SSE协议.md
```

## 风险与取舍

- 不采用过重的企业级 PRD/ADR/RFC 体系，避免学习项目维护负担太大。
- 保留 `接口文档.md` 和既有对接说明，不重复搬运大段接口内容。
- 允许 `NOTE` 更口语化，服务个人学习；`REQ` 和 `DESIGN` 保持相对工程化。

## 验证计划

- 检查新增 Markdown 文件能以 UTF-8 读取。
- 检查 `AGENTS.md` 是否能作为 Agent 入口独立阅读。
- 检查 `docs/progress/PROG-20260722.md` 是否引用本次 REQ/DESIGN。
