# 文档治理规范

本项目采用轻量文档治理。目标不是写很多文档，而是把需求、方案、缺陷、进度和学习总结沉淀下来，让 AI Agent 和人都能在后续任务中快速接上上下文。

## 1. 文档分类

| 类型 | 目录 | 命名格式 | 用途 |
| --- | --- | --- | --- |
| REQ 需求文档 | `docs/requirements/` | `REQ-YYYYMMDD-XX-*.md` | 新功能或大范围改造前记录需求、范围、验收标准 |
| DESIGN 方案文档 | `docs/designs/` | `DESIGN-YYYYMMDD-XX-*.md` | 记录技术方案、交互方案、调用链路、关键取舍 |
| BUG 缺陷记录 | `docs/bugs/` | `BUG-YYYYMMDD-XX-*.md` | 记录 bug 现象、复现、定位、根因、修复、验证 |
| PROG 进度日志 | `docs/progress/` | `PROG-YYYYMMDD.md` | 每天一篇，记录当天完成事项、证据、阻塞和下一步 |
| NOTE 学习笔记 | `docs/notes/` | `NOTE-YYYYMMDD-XX-*.md` | 记录学习总结、链路解释、复盘和大白话理解 |

## 2. 关联规则

- `DESIGN` 必须引用对应 `REQ`，没有 `REQ` 时先补轻量需求。
- `BUG` 如果来自某个需求或改造，必须引用来源 `REQ` 或 `DESIGN`。
- `PROG` 必须引用当天涉及的 `REQ / DESIGN / BUG / NOTE`。
- `NOTE` 可引用任意文档，不强制，但建议标明来源任务。
- 一个任务结束时，应至少更新当天 `PROG`；复杂任务还应补齐 `REQ` 和 `DESIGN`。

## 3. 推荐工作流

用户当前工作流：

```txt
给需求 -> AI 深挖 -> 商榷 -> 清晰方案 -> AI 执行 -> 用户审核 -> AI 修复/用户提交 -> 总结笔记
```

对应落盘方式：

```txt
给需求、深挖、商榷        -> REQ
清晰方案                  -> DESIGN
AI 执行、用户审核、修复    -> PROG
问题定位                  -> BUG
总结笔记                  -> NOTE
```

```md
## 目标

## 范围

## 非目标

## 关联文档
```

### DESIGN 模板

```md
# DESIGN-YYYYMMDD-XX-标题

## 关联需求

## 当前现状

## 设计方案

## 调用链路

## 风险与取舍

## 验证计划
```

### BUG 模板

```md
# BUG-YYYYMMDD-XX-标题

## 来源

## 现象

## 复现步骤

## 修复方案
```


## 5. AGENTS.md 的边界

`AGENTS.md` 只写 Agent 的行为规则、代码边界和文档读取顺序，不写具体需求、接口全文、数据库表结构和历史实现细节。具体知识应放入 `docs/` 对应分类中。
